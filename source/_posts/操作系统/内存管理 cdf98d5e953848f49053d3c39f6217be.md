# 内存管理

Created: January 3, 2022 4:40 PM
Tags: 内存管理

应用程序能够高效安全的使用物理内存的方案: 虚拟内存

程序运行只能使用虚拟地址, CPU负责将虚拟地址翻译为物理地址, 操作系统负责设置虚拟地址和物理地址间的映射.

虚拟内存的目标:

- 高效性:
    - 一方面，虚拟内存抽象不能再应用程序运行过程中造成明显的性能开销
    - 另一方面，虚拟内存抽象不应该占用过多的物理内存资源，从而导致物理内存的有效利用率明显降低
- 安全性:
    - 虚拟内存抽象需要使不同应用程序的内存互相隔离
- 透明性:
    - 虚拟内存需要考虑到对应用程序的透明性，使得应用程序开发者在编程时无需考虑虚拟内存的抽象
    

物理地址： 把物理内存看作一个大数组，每个字节都可以通过与之唯一对应的地址进行运行访问

虚拟地址：物理内存的抽象

地址翻译：将虚拟地址翻译为物理地址

内存管理单元(Memory Management Unit, MMU): CPU的重要部件，负责将虚拟地址转换到物理地址

为了加速地址翻译速度，引入了转址旁路缓存(Translation Lookaside Buffer， TLB)，TLB是属于MMU的内部单元

### 分段 &  分页

MMU将虚拟地址翻译为物理地址的主要机制有两种： 分段机制 和 分页机制

分段：操作系统以“段”的形式管理/分配物理内存

CPU访问虚拟地址中某个段时，MMU会查询段表得到该段对应的物理区域。

虚拟地址 = 段号 + 段内偏移

在翻译过程中：

1. MMU先查询段表基地址寄存器，找到段表的位置，结合段号找到对应段的信息
2. 取出该段的起始地址，加上待段内地址(OFFSET)，找到最终的物理地址

分段机制下，不仅虚拟地址内存空间被划分成不同的段，物理内存也以段位段位进行分配。因此操作系统内购实现物理资源的离散分配，但这也容易导致在物理内存上出现外部碎片，即在段与段之间留下碎片空间(不足以映射给虚拟空间的段，会被浪费掉)

分页：将应用程序的虚拟地址划分成连续的，等长的物理页，同时物理内存也是被划分位连续等长的物理页。按照固定页大小分配物理内存，使得物理资源易于管理，可有效避免分段机制中产生的外部碎片

页表：虚拟地址和物理地址映射的表

虚拟地址 = 虚拟页号 + 页内偏移量 

翻译过程：

1. MMU先解析得到虚拟地址的虚拟页号，并通过虚拟页号去该应用程序的页表 (页表基地址寄存器)中找到对应条目
2. 取出条目中储存的物理页号
3. 最后用该物理页号对应的物理页起始地址加上虚拟地址中的页偏移量就得到 最终的物理地址

为什么需要多级页表?

如果是单级页表,那么页表所占用的空间很大,为了压缩页表大小,引入多级结构的页表.多级页表允许在整个页表结构中出现空洞,而单级页表需要每一项都实际存在.

为什么单级页表中每一页都需要存在?

单级页表可以看成虚拟地址的虚拟页号作为索引的数组,整个数组的初始地址储存在页表基地址寄存器中. 即要求页表是连续的,否则无法根据页表找到对应的物理地址.

在四级页表结构中允许页表内存在“空洞”,假设真个应用程序的内存地址空间中只有两个虚拟页被使用,分别对应最低和最高的两个虚拟地址.在使用4级页表后, 整个页表实际上只需要 1 个 0 级页表，2 个 1 级页表，2 个 2 级页表，2 个 3 级页表，合计 7 个页表页，仅仅占用 28KB 的物理内存空间，远小于单级页表。

为什么硬件仅仅采用简单的 TLB 管理方式，就能在大多数情况下获得较高的 TLB 命中率？

因为局部性起了重要作用，应用程序在运行过程中访问内存的模式具有时间局部性和空间局部性，TLB 中的一条缓存项对应着一个内存,也由于内存访问的时空局部性，TLB 缓存项在将来很可能被多次查询。

如何保证 TLB 中的内容与当前页表内容的一致性？

由于TLB 是使用虚拟地址进行查询的，所以操作系统在进行页表切换（应用程序切换）的时候，需要主动刷新 TLB。若操作系统在切换应用的过程中刷新 TLB，那么应用程序开始执行（被切换到）的时候总会发生 TLB 未命中的情况，进而不可避免的造成性能损失。那么是否能够避免应用程序切换过程中 TLB 刷新的开销呢？

一种为 TLB 缓存打上“标签”的设计正是为了避免这样的开销。AArch64 体系结构为例，它提供了 ASID 功能（x86 64 上对应的功能为 PCID），具体来说，操作系统可以为不同的应用程序分配不同的 ASID 作为应用程序的标签，再将这个标签写入应用程序的页表基地址寄存器的空闲位，同时 TLB 中的缓存项也会包含 ASID 这个标签，从而使得 TLB 中属于不同应用程序的缓存项可以被区分开。因此，在切换页表的过程中，操作系统不再需要清空 TLB 缓存项。

ASID 最多有 16 位，即同时最多可以有 65536 个标签，操作系统需要合理的为应用程序分配标签。

为什么需要这种已分配但未映射到物理内存中的状态？

这种状态出现在换页机制中，当A程序换出(swap out)时，A程序的物理页P（虚拟页为V）被操作系统回收，然后写入到磁盘的一个地方，并且将A程序页表的V-P映射去除，留下V。由此产生已分配但未分配物理内存的状态。

**缺页异常：当应用程序访问分配但未映射到物理内存的虚拟页的时候，就会产生缺页异常。**

**如何处理缺页异常？**

> **通过操作系统提供的缺页异常处理函数，让该函数找到一个空闲的物理页，将 它换入(swap in)，再将之前写入磁盘的数据再加载的物理页中。**
> 

预取机制(prefetching)：当发生换入操作时，预测那些页可能也会被访问，提前将他们一起换入，从而减少缺页的发生

按需页分配(demand paging)：当应用程序申请分配内存的时候，操作系统可选择将新分配的虚拟页标记成已分配但未映射物理内存状态，而不必为这个虚拟

当一个虚拟页处于未分配状态或者已分配但未映射至物理内存状态的时候，应用程序访问虚拟页都会触发缺页异常，操作系统时如何区分的呢？操作系统时如何记录虚拟页分配状态的呢？

Linux 实现方法：由于应用程序通常使用虚拟地址空间的一些区域（比如数据和代码，栈，堆分别对应于三个互不连续的虚拟内存区域），所以在 linux 中，应用程序的虚拟地址空间被实现成由多个虚拟内存区域（VMA）组成的数据结构，每个 VMA 中包含该区域的起始虚拟地址，结束虚拟地址，访问权限等信息，当发生缺页异常是，操作系统判断该虚拟页是否属于该应用程序的某VMA 来区分该页所处的分配状态。

页替换策略：

- MIN/OPT 策略
    - 最优策略 用于衡量其他策略优劣
- FIFO 策略
    - 简单的先到先服务，不管任何信息
- Second Chance 策略：如果在被使用，再次入队(第二次机会)
    - 在FIFO基础上，若队首的页已经被标记为访问，则挪到队尾且清楚标志位，再从队首找淘汰的页
- LRU 策略
    - 最近未访问的页优先被淘汰，list + map实现
- MRU 策略
    - 最近访问的页，有限被淘汰
- 时钟算法策略
    - 指针指向新换入页的页号的后一个，在往后检查页的访问位：
        - 访问位 == false： 直接淘汰
        - 访问位 == true：置为0 继续往后检查

## 虚拟内存

> 虚拟内存抽象使应用能够拥有一个独立且连续的虚拟地址空间，其通过页表和硬件的配合能够在对应用程序透明的前提下自动地进行虚拟地址到物理地址的映射。
> 

**共享内存: 允许同一个物理页在不同的应用程序间共享。**

- 用途：可以让不同的两个程序之间互相通信传递数据

**写时拷贝：页表中每个页表项除了记录物理页号，还记录了其他信息（属性位），用于标识虚拟页的权限（该页是否可写，可执行）的权限位，写时拷贝正式利用标识“是否可写”的权限位来实现的。写时拷贝允许应用 A 和应用 B 以只读的方式共享同一段物理内存，一单某个应用对该内存区域进行修改，就会触发缺页异常（这是由于违反权限导致的，不同于之前所说的换页机制下的未映射导致的）。在触发了缺页异常后，CPU 同样会将控制流传递给操作系统预先设置的缺页异常处理函数，在该函数中，操作系统会发现当前的缺页异常是由于应用程序写了只读内存，而且相应的内存区域又是被操作系统标记成写时拷贝的，于是操作系统会在物理内存中将缺页异常对应的物理页重新拷贝一份，并且将新拷贝的物理页以可读可写的方式重新映射给触发异常的应用程序，此后再恢复应用程序的执行。**

- 建子进程时，将父进程的 `虚拟内存` 与 `物理内存` 映射关系复制到子进程中，并将内存设置为只读（设置为只读是为了当对内存进行写操作时触发 `缺页异常`）。
- 当子进程或者父进程对内存数据进行修改时，便会触发 `写时复制` 机制：将原来的内存页复制一份新的，并重新设置其内存映射关系，将父子进程的内存读写权限设置为可读写。

**内存去重：操作系统定期扫描具有相同内容的物理页，并找到映射到这些物理页的虚拟页，然后只保留其中一个物理页，将具有相同内容的其他虚拟页都用写时拷贝的方式映射到这个物理页，释放其他的物理页。Linux 中的 KSM 实现该功能。但是该功能会对性能产生影响。**

**内存压缩：当内存资源不足时，操作系统选择一些“最近不太会使用”的内存页，压缩其中的数据，从而释放更多空闲内存。如 Linux zswap。**
